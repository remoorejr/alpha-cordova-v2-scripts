# 📘 Alpha Cordova Developer Guide (v2.3.0 Master Edition)

This guide provides a deep dive into the automation architecture of the Alpha Cordova Suite. Version 2.3.0 focuses on **Consolidated Stability**, **Turbo-Sync Performance**, and **Zero-Touch Release Automation**.

---

## 🏗️ Architecture Overview
The suite uses a **Containerized Build Pattern**. By wrapping the Android SDK, Gradle, and Cordova inside a Docker container, we eliminate "it works on my machine" issues caused by local environment drift.

### The "Brain": `release-build.ps1`
The PowerShell engine handles the orchestration between your Windows host and the Linux container. It performs five critical tasks:
1.  **Sanitization:** Preparing the host filesystem for Docker access and aligning home paths.
2.  **Versioning:** Parsing `config.xml`, auto-incrementing `versionCode`, and updating Git tags.
3.  **Changelog Management:** Scraping Git logs to automatically update `CHANGELOG.md`.
4.  **Orchestration:** Passing the correct environment variables and flags to `docker-compose`.
5.  **Deployment:** Interfacing with the host-side `adb` to push signed artifacts to physical hardware.

## 🛡️ The Permission Shield & Security Bridge
One of the primary challenges in WSL2/Docker development is the mismatch between Windows ACLs and Linux UIDs. v2.3.0 implements a dual-layer shield:

### 1. HOME Redirection
We force the container's `$HOME` to the project root (`/home/cordovauser/app`). This prevents the container from trying to write to protected or non-existent system folders in the WSL backend.

### 2. The ".bat" Security Wrapper
To resolve `SecurityError: UnauthorizedAccess` issues for team members, we use Batch wrappers (e.g., `Verify-Environment.bat`). These files use a specific flag to bypass Windows execution policies for the session without requiring system-wide changes:
```batch
powershell -ExecutionPolicy Bypass -File .\release-build.ps1
```

### 3. `.config` Sanitization
Many Node.js tools (like Cordova and Insight) try to write telemetry to `~/.config`. If this folder is owned by Windows "System," the build crashes. The script now force-clears and re-grants `Everyone:F` permissions to a local `.config` directory before every build.

## ⚡ Turbo-Charged Caching (Folder Shadowing)
To achieve near-instant UI updates (<10 seconds), v2.3.0 utilizes **Named Volume Shadowing**. 

### Why it's faster:
Standard "Bind Mounts" (mapping `C:\project` to `/app`) are slow for the thousands of small files generated by Gradle and NPM. In v2.3.0, we map **Named Volumes** directly onto sub-folders *inside* the project root:
* `gradle_cache` -> `/home/cordovauser/app/.gradle`
* `npm_cache`   -> `/home/cordovauser/app/.npm`

**The Result:** Docker "shadows" these folders. The tools believe they are writing to the project directory, but the data is intercepted and stored in the **Linux-native EXT4 filesystem**. This bypasses the slow Windows/WSL file-system bridge entirely.

## 📈 Advanced Release Automation
Version 2.3.0 introduces "Zero-Touch" management for production deployments:

* **Version Bumping:** The engine automatically reads `config.xml`, increments the `version` string (e.g., 2.2.9 -> 2.3.0), and bumps the `android-versionCode` (required for Google Play).
* **Git Log Injection:** During a release build, the script scrapes all commit messages since the last Git tag and prepends them to the top of `CHANGELOG.md` with a datestamp.
* **Automatic Tagging:** Once a build is successful, the script automatically runs `git commit` and `git tag` to lock the release into your repository history.

## 🔊 Audio Notifications (New in v2.3.0)
The build engine now includes an audio feedback system, allowing for "eyes-free" monitoring of the build process.

* **Success Chime (Double High Beep):** Plays when a build or sync completes successfully.
* **Error Alarm (Long Low Tone):** Plays if a build step fails or the container crashes.
* **Prompt Ping (System Asterisk):** Plays when the script requires user input (e.g., version confirmation).

**Benefit:** This allows developers to multitask or step away from their workstations during long compilation tasks, knowing they will be alerted to the final status.

## 🔐 Signing Configuration (`build.json`)
Standardizing on `build.json` is required for the v2.3.0 engine. Unlike `.properties` files, JSON allows the Cordova CLI to accurately parse complex signing arguments for both Bundles and APKs.

**Example `build.json`:**
```json
{
    "android": {
        "release": {
            "keystore": "alpha-release.keystore",
            "storePassword": "your_password",
            "alias": "your_alias",
            "password": "your_password",
            "keystoreType": "jks",
            "packageType": "bundle"
        }
    }
}
```

## 📦 Hybrid Release Strategy
Google Play requires **Android App Bundles (`.aab`)**, but physical devices require **APKs**. v2.3.0 solves this by performing a double-pass build when the `-Install` flag is used:

1.  **Pass 1:** Generates the signed `.aab` for production upload.
2.  **Pass 2:** Generates a signed `.apk` specifically for local testing.
3.  **Deployment:** The script bypasses Cordova's internal `run` logic and uses the **host-side `adb`** to push the signed APK directly. This ensures the app on your phone is bit-for-bit identical to the one going to the store.

## 🔍 Troubleshooting
| Issue | Cause | Solution |
| :--- | :--- | :--- |
| `EACCES: permission denied` | root-owned cache folders. | Run: `docker compose run --rm --user root builder chown -R 1000:1000 /home/cordovauser/app/.npm /home/cordovauser/app/.gradle` |
| `WSL 1 detected` | Incompatible filesystem. | v2.3.0 REQUIRES WSL 2. Run `wsl --set-default-version 2`. |
| `SecurityError` | PS Execution Policy. | Always launch using the provided `.bat` wrapper files. |
| Slow Build Speeds | Cache Mismatch. | Ensure `.gradle` and `.npm` folders are deleted from the Windows host so the Named Volumes can take over. |
